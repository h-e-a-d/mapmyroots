# Phase 1: Astro Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Migrate the existing static MapMyRoots site to Astro without changing what the site does. Marketing pages move into `.astro` files using shared layouts and a single SEO component. The canvas builder app stays a single page that imports its existing JS modules unchanged.

**Architecture:** Astro 5.x in pure static mode (`output: 'static'`). No SSR adapter in this phase — Cloudflare Pages will serve the prerendered output as plain files. CSS migrates from `src/ui/styles/` to `src/styles/`. Static assets at `assets/` move into `public/assets/` so URLs like `/assets/locales/en.json` keep working. The existing `src/core`, `src/ui`, `src/features`, `src/utils`, `src/data`, `src/shapes`, `src/analytics`, `src/config` directories don't move — Astro just imports their entry scripts through Vite.

**Tech Stack:** Astro 5.x, Vite (built into Astro), `@astrojs/sitemap`. The Cloudflare adapter is **not** added here — that comes in Phase 6 when we need SSR. i18n routing, OG image generation, and service worker are **out of scope** (Phases 3 and 4).

**Parent Roadmap:** [`docs/ROADMAP.md`](../../ROADMAP.md) (Phase 1).

**Prerequisite:** Phase 0 must be complete. The repo should be on a clean working tree with all Phase 0 commits landed.

## Out of scope for this phase

These come later — do not attempt them in Phase 1:

- **i18n routing** (`/de/`, `/es/`, `/ru/`) — Phase 3
- **Real OG/Twitter images** — Phase 3 (the broken references stay broken in Phase 1; they were broken before too)
- **Cloudflare adapter / SSR routes** — Phase 6 (deferred)
- **Service worker / PWA** — Phase 4
- **Glossary content collection migration** (markdown sources) — Phase 3
- **Cloudflare Web Analytics swap** (GTM stays in Phase 1) — Phase 2
- **`_headers` / `_redirects` files** — Phase 2
- **GitHub Actions CI** — Phase 2

## Task 2: Create Astro config and TypeScript config

**Files:**
- Create: `astro.config.mjs`
- Create: `tsconfig.json`
- Modify: `.gitignore` (add Astro output dirs)

- [ ] **Step 1: Create `astro.config.mjs`.**

Write the following exact content to `astro.config.mjs`:

```javascript
// @ts-check
import { defineConfig } from 'astro/config';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  site: 'https://mapmyroots.com',
  output: 'static',
  trailingSlash: 'ignore',
  build: {
    format: 'directory',
    assets: '_astro'
  },
  integrations: [
    sitemap({
      filter: (page) => !page.includes('/offline'),
      changefreq: 'weekly',
      priority: 0.7
    })
  ],
  vite: {
    build: {
      sourcemap: true
    }
  }
});
```

Notes:
- `output: 'static'` — pure prerender. No adapter needed.
- `trailingSlash: 'ignore'` — Cloudflare Pages tolerates both forms; let it serve whichever URL the user requests.
- `build.format: 'directory'` — emits `/about/index.html` instead of `/about.html`. Lets us drop the `.html` extension cleanly. (The `_redirects` file in Phase 2 also handles requests for the old `.html` URLs.)
- `build.assets: '_astro'` — hashed bundles go in `dist/_astro/` so we can target them with cache headers in Phase 2.
- The sitemap integration auto-generates `dist/sitemap-index.xml` and `dist/sitemap-0.xml`. The static `public/sitemap.xml` from Phase 0 will be replaced by these (handled in Task 16 below).

- [ ] **Step 2: Create `tsconfig.json`.**

Astro requires a `tsconfig.json` even for JS-only projects. The `astro/tsconfigs/base` preset is permissive enough for our existing JavaScript.

Write the following exact content to `tsconfig.json`:

```json
{
  "extends": "astro/tsconfigs/base",
  "include": [".astro/types.d.ts", "**/*"],
  "exclude": ["dist", "node_modules", "testing", "tests"],
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    },
    "allowJs": true,
    "checkJs": false
  }
}
```

The `@/*` alias lets components import from `src/` cleanly: `import SEO from '@/components/SEO.astro'`.

## Task 6: Create the `SEO.astro` component

**Files:**
- Create: `src/components/SEO.astro`

Single source of truth for `<title>`, `<meta>` tags, `<link rel="canonical">`, OG tags, Twitter tags, and JSON-LD blocks. Pages pass props; this component renders the markup.

- [ ] **Step 1: Write `SEO.astro`.**

Write the following exact content to `src/components/SEO.astro`:

```astro
---
export interface Props {
  title: string;
  description: string;
  /** Path portion of canonical URL, e.g. "/" or "/about". Site origin comes from astro.config.mjs */
  canonicalPath: string;
  /** Optional Open Graph image URL (absolute) */
  ogImage?: string;
  /** Optional Twitter card image URL (absolute) */
  twitterImage?: string;
  /** Stringified JSON-LD blocks; each rendered inside its own <script type="application/ld+json"> */
  jsonLd?: string[];
}

const {
  title,
  description,
  canonicalPath,
  ogImage = 'https://mapmyroots.com/og-image.jpg',
  twitterImage = 'https://mapmyroots.com/twitter-image.jpg',
  jsonLd = []
} = Astro.props;

const canonicalUrl = new URL(canonicalPath, Astro.site).toString();
---

<title>{title}</title>
<meta name="description" content={description} />
<meta name="author" content="MapMyRoots" />
<meta name="robots" content="index, follow" />

<link rel="canonical" href={canonicalUrl} />

<!-- Resource hints -->
<link rel="dns-prefetch" href="https://www.googletagmanager.com" />

<!-- Open Graph -->
<meta property="og:type" content="website" />
<meta property="og:url" content={canonicalUrl} />
<meta property="og:title" content={title} />
<meta property="og:description" content={description} />
<meta property="og:image" content={ogImage} />
<meta property="og:site_name" content="MapMyRoots" />

<!-- Twitter -->
<meta property="twitter:card" content="summary_large_image" />
<meta property="twitter:url" content={canonicalUrl} />
<meta property="twitter:title" content={title} />
<meta property="twitter:description" content={description} />
<meta property="twitter:image" content={twitterImage} />

{jsonLd.map((block) => (
  <script type="application/ld+json" set:html={block}></script>
))}
```

Note: `set:html` is Astro's directive for raw HTML/JSON content, used here so JSON-LD strings are emitted verbatim without escaping.
